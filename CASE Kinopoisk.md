# Кейс: Автотесты для онлайн-кинотеатра «Кинопоиск»

**Роль:** Инженер по автоматизации тестирования (QA Automation)  
**Тип проекта:** Учебный  

---

## Задача

Разработать фреймворк для автоматизированного тестирования ключевого функционала онлайн-кинотеатра «Кинопоиск» - UI (веб-интерфейс) и API.

**Цели проекта:**
- Создать модульную, поддерживаемую архитектуру тестов
- Реализовать UI-тесты с использованием паттерна Page Object Model (POM)
- Настроить гибкий запуск тестов через маркеры pytest

**Ограничения:**
- Тестовое окружение - реальный сайт kinopoisk.ru
- UI-тесты периодически блокируются капчей, автоматический обход невозможен

---

## Процесс решения

### 1. Анализ и архитектура

Перед написанием кода я проанализировала структуру веб-приложения и спроектировала архитектуру фреймворка:

kinopoisk_autotests/

├── config/ # Настройки (URL, селекторы, тестовые данные)

├── pages/ # Page Object Model (базовая и главная страницы)

├── api/ # API-клиент (заготовка для будущих тестов)

├── tests/ # Тесты (разделены по маркерам)

├── conftest.py # Фикстуры pytest

├── pytest.ini # Конфигурация с маркерами ui, api, smoke

└── requirements.txt # Зависимости


### 2. Используемые инструменты

| Категория | Инструменты |
|-----------|-------------|
| Язык | Python 3.9+ |
| Тестовый фреймворк | pytest |
| UI-автоматизация | Selenium WebDriver |
| Паттерн | Page Object Model (POM) |
| API-тестирование | requests (клиент реализован) |
| Отчеты | Allure (настроен) |
| Версионирование | Git + GitHub |

### 3. Реализация

**Что сделано на данный момент:**

1. **Базовая инфраструктура:**
   - Настроен `conftest.py` с фикстурами для WebDriver
   - Создан `pytest.ini` с маркерами (`ui`, `api`, `smoke`)
   - Написаны `settings.py` и `test_data.json` для хранения конфигураций

2. **Page Object Model:**
   - `base_page.py` - базовый класс с общими методами
   - `main_page.py` - методы для работы с главной страницей (поиск, навигация)

3. **Тесты:**
   - Реализован UI-тест поиска фильма (`test_search_film`)
   - Сделан шаблон для API-тестов (`test_api.py`), клиент написан

4. **Гибкий запуск:**
   ```bash
   pytest -m ui -v    # Только UI-тесты
   pytest -m smoke -v # Только smoke-тесты
   pytest -v          # Все тесты


### 4. Выявленные сложности

**Проблема №1: Капча при UI-тестировании**

При запуске UI-тестов на реальном сайте `kinopoisk.ru` периодически появляется капча. Автоматический обход невозможен без нарушения правил сервиса.

**Как это решено на данный момент:**
- Тесты настроены на ручной ввод капчи при её появлении
- Внедрены маркеры pytest - можно запускать только API или только smoke-тесты, минуя проблемные UI-сценарии

**Проблема №2: Отсутствие тестового окружения**

Проект выполняется на реальном продакшен-сайте, что накладывает ограничения:
- Нельзя тестировать деструктивные сценарии (удаление, массовые операции)
- Результаты тестов зависят от внешних факторов (обновления сайта, капча)

**Что можно улучшить в будущем:**
- Договориться о выделении тестового окружения с отключенной капчей
- Либо переориентироваться на API-тесты как основной слой проверки
